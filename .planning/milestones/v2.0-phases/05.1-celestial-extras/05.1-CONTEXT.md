<decisions>
## Implementation Decisions

### Moon (SKY-07 — the design tension, resolved)
- A bright/full moon IS light pollution and would wash out the Milky Way, contradicting the milestone's zero-light-pollution premise. LOCKED: **thin waning crescent**, small (≈3–4% of viewport min-dimension), LOW near the horizon (within the bottom ~30% of sky, above the silhouette line), placed on the OPPOSITE side from the Milky Way band and clear of the content column per 05-UI-SPEC's placement map
- Drawn PROCEDURALLY into Layer 0 during starfield generation (two-circle crescent mask or radial-gradient technique — planner's choice), tokens-only (`--star` tinted, subtle earthshine on the dark limb allowed via low-alpha `--sky-horizon`), NO image assets
- Dim by construction: peak luminance must stay below the Milky Way core's peak (verifiable via the same canvas-readback used by the contrast script)
- Present in the reduced-motion static frame (it's static Layer 0 content)

### Shooting stars (SKY-06)
- Layer 2 transient: at most ONE active meteor at a time; randomized cadence 20–45s; streak lifetime ~0.6–1.0s; short fading trail (gradient stroke along the path — reuse the beam-trail math idiom); trajectory across the UPPER sky only (top ~40%), random angle within a believable range (diagonal downward), never crossing the content column's center or the camper
- Scheduler integrates with the EXISTING pause state machine: no spawns (and any active streak cancelled) when paused {hidden, fig01-active} or `prefers-reduced-motion` (meteors are decorative motion — WCAG C39: absent entirely under reduced motion)
- Zero idle cost: when no meteor is active, the per-frame loop does no meteor work beyond a timestamp check; cadence via the existing frame clock or setTimeout consistent with scene.ts's established patterns (no new rAF)

### Floors (all inherited, re-verified)
- astro check 0 errors; build green; zero hex outside tokens.css (no new tokens expected — reuse --star/--sky-*); fig01 + deck + scene non-regression greps; single-rAF invariant (scene.ts still the only loop)
- Local Lighthouse ≥90 both presets re-run and recorded; idle frame-cost evidence updated
- NO push to origin (Phase 6 owns the deploy); no new dependencies
- Screenshot evidence committed (headless Chrome, both DPRs, one with a mid-flight meteor if capturable — else the streak path drawn in a debug still + code-level criteria)

### Claude's Discretion
- Exact crescent geometry/rotation, earthshine subtlety, meteor angle range/length/velocity, trail gradient stops, cadence RNG shape, whether meteor scheduling lives in scene.ts or a small meteors.ts module

</decisions>

<specifics>
## Specific Ideas

- A meteor should feel RARE and lucky — a visitor who sees one got a moment; a visitor who sees three got a screensaver. Err toward the long end of the cadence
- The crescent reads "late night at a dark-sky site," not "moonlit night" — if in doubt, dimmer and thinner
- Trail fades tip-to-tail with the head brightest (same physics-feel as the fig01 beams)

</specifics>
