# Phase 4 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Source note:** This phase does NOT introduce a new design system. It formalizes the deck CHROME only — panel transition treatment, the mono progress index + jump list, the first-visit hint, the "view classic" link, and focus/announcement states — entirely inside the locked v1 token system (`src/styles/tokens.css`) and the locked v1 UI-SPEC (`.planning/milestones/v1.0-phases/01-foundation-editorial-shell/01-UI-SPEC.md`). Zero new colors, zero new fonts, zero new type sizes. The night-sky scene (scrim, contrast work, ambient motion) is Phase 5 — this contract renders against the plain v1 `--bg` background. Where 04-CONTEXT.md left a gap ("Claude's discretion"), this document makes the call and marks it explicitly.

## Interaction & Motion Contract

*(Supplementary — not a template-required section, but this phase is interaction-heavy and the executor needs unambiguous numbers. Mirrors Phase 1's "Supplementary: Page Structure & Motion" precedent.)*

### Panel transition (Claude's discretion — exact curve/duration; ceiling and easing family locked by 04-CONTEXT.md)

- Duration: **420ms** (inside the locked ≤500ms ceiling, leaves headroom below the ceiling so it never reads sluggish)
- Easing: **`cubic-bezier(0.16, 1, 0.3, 1)`** (a standard "ease-out expo" curve — fast start, long soft settle; matches 04-CONTEXT.md's "measured, ease-out, never rubber-bandy" doctrine and the v1 "one moving thing" premium-restraint feel)
- Motion: incoming panel `translateY(24px) → translateY(0)` + `opacity: 0 → 1`, simultaneously; outgoing panel `translateY(0) → translateY(-24px)` + `opacity: 1 → 0`, same duration/easing, same direction (both slide "up" — advancing feels like paging forward regardless of which direction the input came from, keeping the metaphor simple; reverse navigation uses the mirrored offset, `translateY(-24px) → 0` incoming / `0 → 24px` outgoing)
- Transition lock: input is ignored for the full 420ms while a transition runs (already locked by 04-CONTEXT.md's wheel-accumulation requirement; this duration IS the lock window other input handlers must also respect for consistency across wheel/touch/keyboard/index-click)
- `prefers-reduced-motion: reduce`: duration `0ms`, no transform/opacity animation — panels snap directly to the end state. This is the ONLY branch (no separate "reduced but still animated" tier), per 04-CONTEXT.md and the project's C39-conformant doctrine carried from FEATURES.md ("decorative motion should simply be disabled... meaningful motion should become an instant/static state change that still conveys the information" — a panel change IS meaningful information, so it still happens, just instantly).

### Focus management

- On panel activation: the new active panel's root element (`tabindex="-1"`) receives `.focus({ preventScroll: true })`.
- Because this is a programmatic (non-keyboard) focus move, it must NOT show a visible ring by default (native `:focus-visible` heuristics already suppress this in modern browsers, but declare it explicitly to prevent a stray outline in browsers that focus-ring more aggressively on programmatic `.focus()`):
  ```css
  .panel:focus { outline: none; }
  .panel:focus-visible { outline: 2px solid var(--accent); outline-offset: -2px; }
  ```
  (`outline-offset: -2px`, not the global `+2px`, because the panel is full-viewport — a positive offset would push the ring off-screen.)
- All NEW interactive elements this phase (index pill, jump-list items, view-classic/deck-view link) inherit the existing global `:focus-visible { outline: 2px solid var(--accent); outline-offset: 2px; }` rule from `global.css` — no new focus style needed, this is a confirmation, not a gap.
- Non-active panels: `inert` + `aria-hidden="true"` (locked, 04-CONTEXT.md).

### Progress Index + Jump List (`DeckIndex.astro`) — placement and behavior (Claude's discretion, true gap)

- **Placement:** fixed, bottom-right, `right: clamp(18px, 4vw, 32px); bottom: 58px` (clears the fixed footer bar). Chosen over embedding inside `SiteFooter` to avoid crowding the existing status+clock row and to keep `SiteFooter`'s markup/styling untouched (only its `position` changes, per the "styling-only" architecture note) — the index is a wholly new, wholly separate fixed element.
- **Collapsed (default) state:** a pill showing only the digits, e.g. `03 / 07` — `padding: 6px 12px`, `border: 1px solid var(--hair2)`, `border-radius: 6px`, `background: var(--bg)`. Visually and interactively identical to `.btn` (see Typography/Spacing tables above).
- **Expansion trigger:** hover or focus (desktop/keyboard) OR tap (touch) reveals the jump list — a vertical stack of all 7 panel names, anchored above the pill (grows upward, never off the top of the viewport given its bottom-anchored position), `background: var(--bg)`, `border: 1px solid var(--hair2)`, `border-radius: 6px`, drop no shadow (matches the flat, hairline-bordered surface language used everywhere else — `.fig`, `.tip` — no new elevation system).
- **Jump list items:** one `<a href="#{hash}">` per panel, lowercase-mono panel name (see Copywriting Contract), `padding: 8px 12px` (desktop) / `12px 12px` (≤640px). Current panel gets `aria-current="true"` + a leading `▸` marker + `--accent` text color; all others `--dim` → `--ink` on hover/focus.
- **Keyboard reachability:** the pill and every jump-list item are real focusable elements in DOM tab order (not a CSS-only hover reveal with no keyboard path) — `:focus-within` on the pill's wrapper keeps the list open while any item inside has focus, matching the existing accessible-disclosure pattern already implicit in Fig.01's tooltip-adjacent proxy buttons.
- **First-visit hint placement:** directly above the collapsed index pill (`bottom: 96px`, same `right` alignment), same fixed layer, `opacity` fades 0 over 300ms on dismiss (not an abrupt `display:none` snap — a quiet fade matches the "quiet hint" brief). Positioning it beside the index (rather than center-screen or as a separate corner) ties the hint visually to the control it's teaching, reducing "what is this banner" ambiguity flagged in FEATURES.md's homepage-usability research.
- **Mobile (≤640px):** the jump list, when expanded, becomes a fixed full-width sheet anchored to the bottom (`position: fixed; inset-inline: 0; bottom: 0` sitting above the footer bar) rather than a small floating box — small floating popovers are hard to hit-target on touch; a full-width sheet gives every item its full 44px+ touch height without horizontal cramming. Backdrop: none (no scrim/dim layer this phase — that visual language is reserved for Phase 5's scene-contrast work); dismiss by tapping the (still-visible) collapsed pill again or tapping any item.

### Fixed chrome (SiteHeader / SiteFooter)

- **This phase's call (Claude's discretion, resolving the open item in 04-CONTEXT.md):** SiteHeader and SiteFooter become fixed chrome NOW, in Phase 4, not deferred to Phase 6. Rationale: the deck's 7 panels (`hero` through `contact`) do not include header/footer — they must persist as a stable frame around the panel stack for the deck to be structurally coherent at all (without this, header/footer would either duplicate per-panel or vanish, both wrong). This is a pure `position` change gated under `.deck-active` (no markup change, no new component, no new color) — squarely inside the "styling-only" scope the research note allows.
- Visual treatment is otherwise UNCHANGED — same padding, type, colors as Phase 1. `z-index: 20` for both, above the panel stack (panels are `z-index: 1` or unset/auto within their own stacking context) but below any future modal/overlay layer Phase 5+ might introduce.
- Under the v1 progressive-enhancement fallback (no JS / `.deck-active` never applied), header/footer render exactly as Phase 1 shipped them — static, in-flow, no fixed positioning. This is the default state; fixed positioning is strictly additive and scoped under `.deck-active`.

### Motion budget note (forward-looking, informational only — not built this phase)

Per FEATURES.md's cross-phase motion-budget warning: this phase's ONLY motion sources are the panel transition itself and the hint's dismiss-fade. Both are user-driven (fire on input), not ambient/idle. This leaves the full "one moving thing per viewport" ambient-motion budget untouched for Phase 5 (starfield/fireflies/constellation-brighten) to spend — Phase 4 introduces zero idle/looping animation.
