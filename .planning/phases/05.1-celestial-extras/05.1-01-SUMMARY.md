## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 - Bug] MASK_OFFSET_RATIO 1.85 → 0.25 (crescent geometry)**
- **Found during:** Task 1
- **Issue:** With the locked MASK_RADIUS_RATIO 1.05, the spec table's offset 1.85R puts the mask's left edge at +0.80R — producing a NEAR-FULL disk with a 0.20R dark bite on the right, contradicting the same spec section's locked visual ("thin left-edge crescent ~0.20×R at its widest, left-facing C") and SKY-07 itself ("thin waning crescent"; a full moon IS light pollution per 05.1-CONTEXT).
- **Fix:** offset 0.25R — the unique value satisfying lit-width = R + offset − maskR = 0.20R. Derivation documented in a code comment at the constant.
- **Files modified:** src/lib/nightsky/starfield.ts
- **Commit:** f783638 (screenshot evidence confirms the correct thin waning-crescent read at both widths)

**2. [Rule 3 - Blocking] DPR2 capture path swapped to the CLI screenshot flag**
- **Found during:** Task 3
- **Issue:** CDP `Page.captureScreenshot` hangs indefinitely whenever deviceScaleFactor 2 is active (via `Emulation.setDeviceMetricsOverride` OR `--force-device-scale-factor=2`) in this environment's headless Chrome — repeated timeouts with scripts frozen and GPU disabled.
- **Fix:** Used the plan's own sanctioned alternative — `chrome --headless=new --screenshot --force-device-scale-factor=2` with a virtual-time budget long enough for the chunked Layer-0 queue to drain (nondeterministic; retried until the painted frame was verified visually). DPR1 + reduced-motion stills used normal CDP capture.
- **Files modified:** none (tooling-only)
- **Commit:** 2fcf6ab

None of the deviations changed any locked parameter other than the geometrically-broken mask offset.
