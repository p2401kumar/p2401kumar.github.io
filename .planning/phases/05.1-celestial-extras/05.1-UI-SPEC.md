## MOON (SKY-07)

### Placement — derives from the Milky Way's side, per 05.1-CONTEXT.md

The Milky Way's bright core sits in the **right** margin (05-UI-SPEC: core through `x:0.80–0.98`, entering top `x:0.55–0.65`). The moon is therefore locked to the **left margin**, using the identical column-edge formula 05-06 established for margin-aware placement (`contentColumnEdges` in `scene.ts` — mirror it into `starfield.ts`, do not import across the nightsky/fig01-style module boundary that already governs this codebase):

```
pad = clamp(18, cssWidth * 0.04, 32)
half = min(880, cssWidth - 2*pad) / 2
columnLeft = cssWidth/2 - half   // px, left edge of the 880px content column
```

| Constant | Value |
|---|---|
| `MOON_RADIUS_COEFF` | `0.018` |
| `MOON_RADIUS_FLOOR` | `12px` |
| `MOON_RADIUS_CAP` | `22px` |
| `MOON_X_MARGIN_FRACTION` | `0.30` |
| `MOON_Y_FRACTION` | `0.68` (of `cssHeight`) |

**Formulas:**

- `R = clamp(MOON_RADIUS_FLOOR, MOON_RADIUS_COEFF * min(cssWidth, cssHeight), MOON_RADIUS_CAP)` — targets ≈3.6% of min-dimension at both check widths (1440×900 → 16.2px radius = 3.6% of 900; 1280×800 → 14.4px → clamped floor territory, ≈3.5% of 800), matching 05.1-CONTEXT.md's locked "≈3–4% of viewport min-dimension." The 12px floor is a documented legibility exception below the reference band (same pattern as 05-UI-SPEC's own camper `clamp(160px,16vw,260px)` width exception) — sub-650px min-dimension viewports may read closer to 5–7%, not a regression.
- `moonX = MOON_X_MARGIN_FRACTION * columnLeft` (px). **At 1440:** `columnLeft=280.8px → moonX≈84.2px` (fraction `0.0585`). **At 1280:** `columnLeft≈199.7px → moonX≈59.9px` (fraction `0.0468`).
- `moonY = MOON_Y_FRACTION * cssHeight` (px) — fraction `0.68`, inside 05.1-CONTEXT.md's locked "bottom ~30% of sky above the silhouette line" band (`y: 0.595–0.85`, derived from the horizon at `y:0.85` per 05-UI-SPEC), and specifically positioned to clear both neighbors it shares the left margin with: above the camper's roofline (camper top ≈ `y:0.71` at reference size, per 05-UI-SPEC's `.camper { bottom:15%; clamp(160,16vw,260)px wide; aspect-ratio 200/110 }`) and below the `samsung` constellation cluster's lower bound (`y:0.65`, 05-UI-SPEC placement table) — roughly 0.03 clearance each side.

**Margin-clearance check (the "check 1280/1440 like 05-06 did" requirement):** `moonX + R + 80px (COLUMN_EDGE_RAMP_PX, the existing sky-brightness-governor ramp width)` must stay left of `columnLeft`. At 1440: `84.2+16.2+80=180.4 < 280.8` (≈100px spare). At 1280: `59.9+14.4+80=154.3 < 199.7` (≈45px spare). Both pass — the moon never falls inside the governed column or its smoothstep ramp at either check width, so `columnAttenuation()`/`starAlphaCapAt()` (starfield.ts's existing governor functions) are **never applied to the moon** — its own fixed alpha ceiling (below) is its entire brightness governance; no changes to those two functions are needed or permitted.

**Constellation-cluster note (non-gated, discretion zone):** `microsoft`/`samsung` also occupy the left margin post-05-06-remap. CONTEXT.md is silent on moon-vs-constellation separation, so this is not a hard gate — but include a visual check in this phase's required screenshot evidence (both check widths) confirming the moon reads as a distinct element. If visually cluttered against a cluster, nudging `MOON_X_MARGIN_FRACTION` to `0.15–0.20` is a same-file constant tweak, not a spec revision.

### Crescent geometry — two-circle offset technique

| Constant | Value |
|---|---|
| `MASK_RADIUS_RATIO` | `1.05` (mask circle radius = `1.05 × R`) |
| `MASK_OFFSET_RATIO` | `1.85` (mask center offset = `1.85 × R` from the lit circle's center, along `+x`) |
| `LIT_ALPHA` | `0.45` (range `0.40–0.50` if tuning is needed; **hard ceiling `0.55`** — see Luminance Ceiling below) |
| `EARTHSHINE_ALPHA` | `0.08` (range `0.06–0.10`) |

**Construction (planner's choice of exact canvas API — destination-out compositing or a two-arc lune path both satisfy this; the geometry is what's locked):**

1. Earthshine (drawn first, full disk): circle at `(moonX, moonY)`, radius `R`, fill `rgba(--sky-horizon, EARTHSHINE_ALPHA)`.
2. Lit crescent (drawn second, on top): the region of a circle at `(moonX, moonY)` radius `R`, minus a circle at `(moonX + MASK_OFFSET_RATIO*R, moonY)` radius `MASK_RADIUS_RATIO*R`, fill `rgba(--star, LIT_ALPHA)`. The mask offset toward `+x` (rightward) carves a thin sliver on the **left** edge of the disk (~`0.20×R` wide at its widest) — a left-facing "C" silhouette, the conventional waning-crescent read locked in 05.1-CONTEXT.md ("thin waning crescent... if in doubt, dimmer and thinner").
3. No rotation applied — axis-aligned, matching the reference geometry above.

Draw order within `generateLayer0`: **after** the Milky Way composite (last Layer 0 primitive) — the moon never interacts with MW pixels (disjoint margins) so order has no correctness effect, but is fixed here for implementation clarity.

### Luminance ceiling (verifiable via canvas readback)

`LIT_ALPHA` must never exceed **0.55** — strictly below the Bright star band's own ceiling (`1.0`, 05-UI-SPEC magnitude table) and far below the Milky Way core's typical composited pixel value outside the governed column (05-06's SUMMARY records ungoverned MW dust as "saturated... up to full white" via `'lighter'` accumulation). This satisfies 05.1-CONTEXT.md's "peak luminance must stay below the Milky Way core's peak" by construction, but must still be **measured, not assumed** — extend `scripts/verify-contrast.mjs` (or a small companion script, planner's choice) with a two-point canvas-readback assertion:

- Sample point A: the moon's brightest composited pixel (center of the crescent's thickest point, `(moonX - 0.5R, moonY)` approximately).
- Sample point B: a known-bright MW core pixel, e.g. `(0.90 × cssWidth, 0.50 × cssHeight)`.
- Assert luminance(A) < luminance(B) at both 1280×800 and 1440×900.

### Interaction with the sky-brightness governor and content column

Already covered by the margin-clearance check above — the moon sits entirely outside `COLUMN_HALF_WIDTH_PX + COLUMN_EDGE_RAMP_PX` at both check widths, so it never falls under any text region and is exempt from `starAlphaCapAt`. `verify-contrast.mjs --cdp` at 1280×800 and 1440×900 **must be re-run after implementation** and must not regress below the current recorded baseline (8.24:1 at 1440, 8.64:1 at 1280, floor 4.5:1, per 05-06's `contrast-evidence.md`). Any regression is a Rule-2 defect requiring the same governor-style fix pattern 05-06 established (attenuate/reposition at the source — never just widen the scrim).

### DPR handling

Automatic — the moon draws inside `generateLayer0`'s existing `layer0Ctx`, already `setTransform(dpr,0,0,dpr,0,0)`-transformed (starfield.ts module header). All moon coordinates/radii above are plain CSS-pixel values, identical convention to every other Layer 0 primitive (stars, MW dust). No separate DPR math, no new canvas, no new context.

### Reduced motion

Present unchanged in the reduced-motion static frame — the moon is baked into Layer 0, which is already static regardless of motion preference (05-UI-SPEC's Reduced-motion table, Layer 0 row: "Unchanged — already static").

## METEOR (SKY-06)

### Parameters

| Constant | Value |
|---|---|
| Active meteors | At most **1** at a time (locked) |
| Cadence | `20000 + (Math.random() ** 0.6) * 25000` ms — range `[20000, 45000]`ms, skewed toward the long end (exponent `<1` biases the distribution upward) per 05.1-CONTEXT.md's "err toward the long end" |
| Life | `life = uniform(0.6, 1.0)` seconds (locked range, uniform — no skew specified) |
| Path length | `pathLength = uniform(0.12, 0.20) * cssHeight` px |
| Velocity | `velocity = pathLength / life` px/s — **derived**, not a constant; varies per spawn from the two independent randoms above |
| Angle from horizontal | `angle = uniform(20°, 35°)`, tilted toward the outer edge (away from the column) |
| Head radius | `2.0px` (range `1.8–2.4px`) |
| Head peak alpha | `0.95` (fixed; never exceeds the `--star` token's own max) |
| Trail length | `uniform(0.06, 0.10) * cssHeight` px — shorter than `pathLength`, so the fading tail visibly trails a moving head across a longer empty path (comet look, not a static bar) |
| Trail stroke width | `1.4px` (range `1.2–1.6px`) |
| Color | `--star` token exclusively — head fill and trail gradient both |

**Trail gradient stops** (linear gradient along the stroke direction, head end → tail end, tip-to-tail fade, head brightest per 05.1-CONTEXT.md's Specifics "same physics-feel as the fig01 beams"):

| Stop | Alpha |
|---|---|
| 0% (head, seamless join to head fill) | `0.95` |
| 45% | `0.45` |
| 80% | `0.12` |
| 100% (tail) | `0` |

### Spawn zone & exclusion construction

Meteors are confined **entirely to the horizontal margins outside the content column** — the same formula and doctrine 05-06 established for fireflies (`contentColumnEdges`/margin containment), deliberately stronger than the literal 05.1-CONTEXT.md wording ("never crossing the content column's center") because the accessibility note explicitly wants meteors "outside text regions" as a placement guarantee, not just off-center. This automatically satisfies all three exclusion zones the orchestrator's brief named:

- **Content column center band:** satisfied — the path never enters the column's x-range at all (stronger than avoiding just the center).
- **Camper region:** satisfied by construction — path y stays ≤ ~0.33×cssHeight-equivalent-sky-fraction (upper 40% of the locked 0–0.85 sky band = `y ≤ 0.34`), while the camper occupies `y≈0.70–0.85` (05-UI-SPEC) — a ≥0.36 vertical gap, no separate collision check needed.
- **Bottom 60% of viewport:** satisfied by construction — the same upper-sky y-bound (`≤0.34`) is well clear of `y>0.4`.

**Construction (per spawn attempt):**

1. Compute `marginLeftPx = columnLeft`, `marginRightPx = cssWidth - columnRight` (mirror `contentColumnEdges`, same as the moon section).
2. If both `< 60px`: **skip this spawn**, reschedule with a fresh random delay (narrow-viewport fallback — meteors simply don't appear on very narrow viewports; documented trade-off, not a regression).
3. Else pick a side: 50/50 if both margins `≥ 60px`; otherwise use whichever side qualifies.
4. **Left side:** `spawnX = uniform(16, marginLeftPx - 12)` px (12px buffer off the column edge, matching the codebase's existing `FIREFLY_MARGIN_CUSHION`-style buffer convention). Direction of travel: down-and-**left** (further from the column) — angle applied as `dx = -pathLength·cos(angle)`, `dy = +pathLength·sin(angle)`. Because travel moves strictly away from the column, the path can safely extend past the viewport's left edge (clipped naturally by the canvas) without ever re-entering the column — no upper bound on `dx` magnitude is needed.
5. **Right side:** mirrored — `spawnX = cssWidth - uniform(16, marginRightPx - 12)`, travel down-and-**right**.
6. `spawnY = uniform(0.03, 0.20) * cssHeight`. Worst-case path bottom: `spawnY_max + pathLength_max·sin(angle_max) = 0.20·cssHeight + 0.20·cssHeight·sin(35°) ≈ 0.3148·cssHeight < 0.34·cssHeight` — holds at any aspect ratio since both terms scale with `cssHeight`.

### Scheduling (setTimeout, zero new rAF)

- Owned by a small **`meteors.ts`** module (recommended, mirroring `constellations.ts`'s shape: its own state + an `advance(ts)`/`draw(ctx,...)` hook called from `scene.ts`'s existing `drawFrame`, plus its own `setTimeout`-based spawn scheduler) — a recommendation, not binding; CONTEXT.md leaves the file split at Claude's discretion.
- Next spawn is scheduled **spawn-to-spawn** (from the previous spawn's timestamp, not the previous meteor's completion) using the cadence formula above.
- At each scheduled fire: gate-check against the **existing** `shouldRun` state (`tabVisible && !fig01Active && !rm.matches`, `scene.ts`'s `updateRunState` condition) — if false, skip the spawn and reschedule with a **fresh** random delay (never an immediate retry, avoiding a pent-up queue after a long pause).
- If a fire somehow lands while a meteor is still active (should never occur given cadence `≥20s ≫` life `≤1s`), skip and reschedule — defensive only.
- Pause mid-flight: an active meteor is **immediately discarded**, mirroring constellations.ts's firing-suppression precedent ("suppression also discards any in-flight beam") — same treatment, applied here.
- Reduced motion: meteors **never spawn** (WCAG C39, absent entirely — not dampened, per 05.1-CONTEXT.md and 05-UI-SPEC's Reduced-motion doctrine). If the OS toggles reduced-motion mid-flight, the active meteor is cleared instantly via the same `onMotionChange` re-branch pattern `scene.ts` already uses.

### Per-frame cost

The `drawFrame` tick gains exactly one new branch: `if (activeMeteor) { advance + draw }` else no-op. When idle (no active meteor, the overwhelming majority of frames given the 20–45s cadence vs ≤1s life), the cost is a single reference null-check — no measurable regression to 05-06's `frame-cost-audit.md` baseline (~70 typical draws/frame) is expected; re-measure as part of this phase's gate battery, don't assume.
