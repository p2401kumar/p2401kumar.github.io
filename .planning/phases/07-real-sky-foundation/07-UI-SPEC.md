# Phase 7 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **This is an ADDENDUM to `05-UI-SPEC.md` (night-sky scene) and `05.1-UI-SPEC.md` (moon + meteor), not a replacement.** Every token, layer split, motion contract, scrim, star/constellation/moon/meteor geometry already locked there applies unchanged unless explicitly superseded below. This document governs ONLY what changes when the procedural Milky Way sky-wash is replaced by a real composited photo: crop/placement, tone/exposure/grading, the ground seam, overlay-vs-photo visual harmony, LQIP appearance, and the new photo-credit copy. Glass (Phase 8) and ambient systems (Phase 9) are explicitly out of scope. See **Locked inheritance** at the end for the full carry-forward list.
>
> **Scope boundary:** this document does NOT set AVIF/WebP quality, bitdepth, or effort settings — that is Spike 1's job (`SPIKE-BANDING.md`, the winning encode recipe). This document governs what the crop LOOKS like and WHERE it SITS, assuming whatever encode recipe the spike locks.
>
> **Reference viewports:** 1440×900 and 1280×800 (matching every prior phase's check-width pair). All placement numbers are stated as viewport fractions first, reference-pixel values second.

## Crop + Placement Contract

The target: the photo's real Milky Way band must land where the procedural band already lived, so returning visitors read this as "the same campsite on a clearer night," not a redesign.

### Band geometry to reproduce (source of truth: `starfield.ts`'s spike-locked constants + `05-UI-SPEC.md`)

The procedural band's centerline ran from `x: 0.63` (top edge, `y:0`) to `x: 0.87` (at the horizon line, `y: 0.85` of viewport height) — a diagonal of **~24° from vertical**, inside the locked 20–25° range. The photo's rotate/crop must reproduce this same diagonal, at the same reference fractions:

| Viewport | Top-entry point | Horizon-exit point | Core (brightest quartile) x-range |
|---|---|---|---|
| 1440×900 | `x≈907px` (`0.63`), `y:0` | `x≈1253px` (`0.87`), `y≈765px` (`0.85`) | `x: 0.80–0.98` (`1152–1411px`) |
| 1280×800 | `x≈806px` (`0.63`), `y:0` | `x≈1114px` (`0.87`), `y≈680px` (`0.85`) | `x: 0.80–0.98` (`1024–1254px`) |

The core tapers to near-zero brightness by `x: 0.65` on its inner (left) edge — same taper rule 05-UI-SPEC already locked for the procedural version.

### Rotation

Rotate the source equirectangular master (`noirlab2430b`, no fixed horizon — any rotation is equally physically valid per `IMAGERY.md`) so the Milky Way plane runs at **~24° from vertical**, matching the diagonal above. Fill any exposed rotation-corner with `#0f1216` (`--bg`, via `sharp().rotate(angle, { background: '#0f1216' })`), never transparent/white, so any corner spill blends with the page rather than showing a seam.

### Master authoring

- Author ONE working master at a **wide letterbox aspect** (recommend ~2.3:1–2.4:1, e.g. crop the rotated source to ~5760×2450 before downscaling) — deliberately wider than any expected viewport aspect (mobile portrait ~0.45–0.6 up to ultrawide ~2.4–3.4) so `object-fit: cover` never has to reveal beyond the master's real content in a way that relocates the band.
- Deliver at **2560w + 1920w** (locked, `07-CONTEXT.md`), plus a **~32px-wide inline LQIP** (see LQIP Appearance). Format/quality/bitdepth: Spike 1's output, not this document.
- Do the rotate + crop **once**, by hand or one-off script, and commit the result (`src/assets/sky/` or similar) — never re-derive at build time on every Astro build. Raw TIFF source stays gitignored, not checked in (`07-CONTEXT.md`).

### `object-fit` / `object-position` — the responsive ladder

Single master, `object-fit: cover`, full-bleed (`position: fixed; inset: 0`, same host as the canvas, sitting BELOW it in DOM order — see z-order below). `object-position` is tuned per breakpoint so the SAME master serves 360px→4K without a second art-directed asset:

| Breakpoint | `object-position` | Rationale |
|---|---|---|
| `< 640px` (narrow mobile) | `12% 25%` | Anchors to a QUIET, low-band-density region of the master, deliberately AWAY from the vivid core. At this width the content column spans nearly the full viewport (05-06's governor doctrine: "text spans the full width" at narrow viewports) — showing the bright galactic core here would repeat the exact failure mode the SKY-05 governor fixed for procedural stars, and a static raster cannot be dynamically dimmed under text. Steering the crop window away from the band entirely is the static equivalent of that governor. |
| `640–1023px` (tablet / narrow laptop) | `55% 32%` | Transitional — margins exist but are still narrow; shows a hint of the band entering from the right without pulling the brightest core fully into frame. |
| `1024–1799px` (desktop — the REFERENCE tier the table above was derived for) | `72% 38%` | The hero composition: band core lands in the right margin per the target table. |
| `≥ 1800px` (ultrawide) | `78% 34%` | Nudged further right/up so the extra revealed width doesn't pull the crop window's left edge into an unintended part of the master. |

**Ultrawide safety invariant (why the band can't drift into the content column):** the content column is a FIXED 880px centered block (`deck.css`), not viewport-relative — as viewport width grows, the column occupies a *shrinking* fraction of the frame (e.g. at 2560px wide, column edges sit at `x: 0.31–0.69`, well inside the `0.80–0.98` core zone with enormous margin to spare). The risk is exclusively at the NARROW end, which the `<640px`/`640–1023px` tiers above handle by steering the crop away from the band, not by fighting a shrinking margin.

### z-order (restated for this document's scope, per `07-CONTEXT.md`)

Photo `<img>`/CSS background (bottom, full-bleed, DOM child of `.nightsky-host` BEFORE the canvas) → overlay canvas (`#nightsky-canvas`, now sky-wash-transparent, draws only stars/moon/constellations/meteors/ground-fill) → camper SVG (Layer 1, unchanged) → content/glass (Phase 8). `.nightsky-host` keeps its `z-index: -1` doctrine unchanged (see `NightSky.astro`'s existing stacking comment) — the photo simply becomes a new first child inside that same negative-z host, never a sibling with its own z-index.

## Tone Contract

The photo must read as an upgrade to the SAME dark palette, not a second, competing look.

### Black point (max ΔL so the seam blend disappears)

Grade the master's shadow floor — non-galactic dark-sky regions, away from any star or dust — to **RGB ≈ (5, 7, 10) ± 4 per channel**, i.e. matching `--sky-zenith` (`#05070a`) directly. This is the SAME darkest tone the procedural gradient already established at the top of frame; a photo whose "black" reads lighter or warmer than this would visibly clash the instant it's compared against the still-drawn ground fill / camper silhouette (both `--sky-zenith`).

### Near-horizon ceiling (non-band regions)

The photo's natural atmospheric glow near the bottom of the sky crop (before the fade-to-`--bg` ramp, roughly `y: 0.75–0.85`, AWAY from the band) must not exceed `--sky-horizon` (`#141a2c`, RGB `20, 26, 44`) in brightness — this is an upper ceiling, not a target match; real photo texture can vary underneath it.

### Exposure ceiling under text (replaces the procedural column governor)

The procedural governor (`columnAttenuation`/`starAlphaCapAt` in `starfield.ts`) attenuated Milky Way dust/haze to `0.12×` alpha and capped star alpha at `0.25` inside the content column (`±464px` of center + `80px` ramp) — because a live per-pixel governor cannot run against a static raster, this duty moves to **build-time grading**: bake an equivalent darkening into the master itself.

- **Technique:** composite a masked darken pass (toward `#05070a`, `--sky-zenith`) over the master, using the IDENTICAL column geometry the procedural governor used — half-width `464px` (at the 1440×900 reference), `80px` smoothstep ramp, scaled proportionally for the 1920w/2560w export widths. Full darken strength inside the column (target: visually equivalent to the procedural `0.12×` alpha reduction — i.e. any band texture under the darkened zone should read essentially flat/near-zero-visible), ramping to zero darkening (full photo brightness) outside the ramp.
- **Floor to hold:** the composited worst-case pixel under real text (photo + column vignette + the existing `.deck::before` scrim, sampled via this phase's minimal photo-aware extension to `verify-contrast.mjs`, per `07-CONTEXT.md`) must match or beat the SKY-05 baseline — **worst-case ≥ 4.5:1 vs `--ink`**, targeting the same order of headroom the procedural fix achieved (`8.24:1`/`8.64:1`). This is a hard gate, not a suggestion — do not silently ship a shortfall (same doctrine `05-06-SUMMARY.md` already proved out).
- **Never** implement this as a runtime CSS `filter`/`backdrop-filter` darken — it is a ONE-TIME build-time pixel bake into the master, per the explicit constraint in this phase's brief and `PITFALLS.md`'s standing "never `filter: blur()`/live blur" doctrine, extended here to "never a live darkening filter" for the same performance/precedent reasons.

### Saturation bounds

Real astrophotography frequently carries warm airglow amber, faint green airglow bands, or reddish nebulosity flecks the procedural palette never had. These must not introduce a second, competing hue family against the site's cool blue-slate graphite + single copper accent:

- Grade the master's bulk dust/haze/background sky toward the SAME blue-slate hue family `--bg`/`--hair`/`--hair2`/the sky tokens already establish (hue range **200–230°**, blue-channel-dominant) — target overall saturation **≤ 15%** (HSL S) in non-star background pixels.
- Individual bright stars may keep minor natural color variation (comparable to `starfield.ts`'s own `jitterStarColor` precedent — small, mostly-neutral, occasional cool/warm drift, never a strong saturated hue).
- If the chosen crop window naturally contains a visibly green/red anomaly (airglow band, nebulosity patch) inconsistent with the site's palette, either re-choose the crop to avoid it or desaturate that specific region in the same build-time grading pass — do not ship a crop with a visible clashing color patch.
- All grading is a ONE-TIME build-time adjustment (`sharp` levels/HSL/tint operations on the master) — never a runtime CSS filter.

## Seam Contract

### Bottom gradient-blend (locked, `07-CONTEXT.md`: "bottom gradient-blend to `--bg`")

Bake a vertical alpha blend into the master, from pure photo content down to solid `--bg` (`#0f1216`):

| Stop (fraction of crop height) | State |
|---|---|
| `0 – 0.72` | Pure graded photo content, no blend |
| `0.72 – 0.94` | Linear (or gentle-ease) alpha ramp from photo → `#0f1216` |
| `0.94 – 1.0` | Fully solid `#0f1216` |

**Why the ramp starts well above the horizon line (`y:0.85`), not at it:** Layer 0's canvas generation is idle-scheduled/chunked and can take a perceptible moment after the photo (LCP-critical, preload-discovered) has already painted. During that gap there is no canvas ground fill covering the photo's lower portion yet — the photo's OWN bake-in blend is the defensive fallback that keeps the composition looking finished before the canvas catches up, not an optional nicety.

### Ground-fill consistency (Rule-2-style catch — preempt this before it becomes a visible seam)

The current procedural ground fill (`starfield.ts`, the flat rect below `horizonYCss`) uses `--sky-zenith` (`#05070a`), while this phase's photo blend target is `--bg` (`#0f1216`) — two visually close but DIFFERENT dark tones. If the canvas ground-fill rect is kept as-is, a faint seam will appear at the horizon boundary between the photo's blended-to-`--bg` bottom and the canvas's `--sky-zenith` ground rect above it.

**Resolution (planner's call, either is acceptable, but pick one deliberately — do not leave both colors in play unexamined):**
1. **Preferred:** remove the canvas's flat ground-fill rect this phase. Since the photo already blends to `--bg`, and `--bg` is the page's own root background sitting beneath the fixed `z-index:-1` `.nightsky-host`, the ground reads correctly with nothing extra drawn — this also satisfies `07-CONTEXT.md`'s "generation gets cheaper" note for Layer 0.
2. **Alternative:** keep a ground-fill rect but change its fill to `--bg`, not `--sky-zenith`, so it matches the photo's blend target exactly.

Either way, the camper silhouette (`--sky-zenith` fill, unchanged SVG) sits on top and is unaffected — it was always a slightly-darker silhouette against whatever ground tone sits behind it; that relationship doesn't change.

### Horizon/silhouette/camper continuity

Ground treatment is otherwise **unchanged** — the camper SVG (Layer 1), its copper glow, its `bottom:15%`/`left:8vw`/`clamp(160px,16vw,260px)` geometry, all carry forward verbatim. The photo is sky-only (no terrain in the equirectangular master); the composite crops sky only, and the site's existing drawn ground/camper layer supplies 100% of the "ground" read, exactly as before.

## Overlay Harmony

The drawn elements must stay unmistakably "the brightest, authored things on the page" against a photo now full of real, dense, un-authored stars.

### Constellation stars vs. photo star noise

- **Radius floor raised within the existing range:** use the UPPER half of 05-UI-SPEC's `2.5–4px` ambient-star range — **3.0–4.5px** for constellation stars specifically (ambient-field stars, now reduced to the ~40-star twinkle subset, are unaffected and keep their original 05-UI-SPEC bands). A delivered 2560w photo's individual real stars typically resolve to ~1–2px specks after resize — a 3px+ drawn star reads as categorically larger at every check width.
- **NEW: subtle ambient halo.** 05-UI-SPEC's alpha-state table gave the "Ambient" (resting) constellation state `none` for halo — correct when the backdrop was a governed-dark procedural sky, insufficient now against a busier real photo. Add a faint halo to the Ambient state: radius **≈1.3× star radius**, alpha **0.12–0.18** (a dimmer sibling of the existing Brightened-state halo: `1.5×` radius, `0.20–0.30` alpha — do not reuse that exact value, keep Ambient visibly quieter than Brightened). Dimmed state stays `none` (unchanged) — it is deliberately meant to recede.
- **Link stroke: unchanged** (`0.75–1px` hairline). A straight geometric line segment is a shape no real starfield produces — it remains legible against any photo density without modification.

### Drawn moon vs. photo Milky Way core

`LIT_ALPHA` stays **0.45** (hard ceiling **0.55**, unchanged from `05.1-UI-SPEC.md`) — no change to the moon's own formula. What changes is the COMPARISON target: the moon must read dimmer than the PHOTO's brightest galactic-core pixel (previously the procedural MW's brightest pixel). This is exactly the physical-honesty story `IMAGERY.md`/`07-CONTEXT.md` already lock in ("rejecting the photo moon is a STORY... keep the decision note") — a moon bright enough to compete with a vivid Milky Way is not physically plausible in one real exposure, so the drawn crescent must stay visibly the dimmer of the two.

Extend `05.1-UI-SPEC.md`'s two-point canvas-readback assertion to the composited photo:
- Sample point A: the moon's brightest composited pixel (unchanged formula, `05.1-UI-SPEC.md`).
- Sample point B: the photo's brightest galactic-core pixel at the equivalent margin position (e.g. `x: 0.90 × cssWidth`, `y: 0.50 × cssHeight`, both check widths) — this is the "minimal photo-aware sampling" `07-CONTEXT.md` calls for, pulled forward into this phase (the full screenshot-sampling re-architecture stays a Phase 8 gate item).
- Assert `luminance(A) < luminance(B)` at both 1280×800 and 1440×900. Any failure is a Tone Contract exposure-ceiling defect, not a moon defect — fix by darkening the photo grade, never by brightening the moon past its `0.55` hard ceiling.

### Meteors

Unchanged geometry (`05.1-UI-SPEC.md`, all constants carried verbatim — head radius `1.8–2.4px`, peak alpha `0.95`, `--star` color, confined to margins `y ≤ 0.34H` outside the content column). Confirmed no-change: meteors already draw at a higher alpha than any ambient star AND are the only moving point of light in the margins at any given moment — they remain trivially distinct from a static photo's real stars regardless of the photo's own density.

## LQIP Appearance

- **Generation:** a ~32px-wide (locked, `07-CONTEXT.md`) heavily downsampled + blurred derivative of the SAME graded master (`sharp().resize(32).blur()` at build time), inlined as a base64 `background-image` (or the `<img>`'s own tiny placeholder `src`, swapped) — no network round-trip for the placeholder.
- **Appearance:** because it's a downsample of the already-graded master, it inherits the same dark, blue-slate, `--bg`/`--sky-zenith`-adjacent tone automatically — no separate tinting step needed. It reads as an out-of-focus, correctly-toned smear of the same sky, never a mismatched flash of unrelated color.
- **Sizing (zero CLS):** the placeholder and the full-resolution image share IDENTICAL `object-fit`/`object-position` values and an explicit reserved box (via `width`/`height` attributes or `aspect-ratio`, full-bleed `inset:0` regardless) — the box is correctly sized before either image paints, so no layout shift occurs at any point in the swap.
- **Transition:** fade the full-resolution `<img>` in via `opacity: 0 → 1` over **400ms**, `cubic-bezier(0.16, 1, 0.3, 1)` — the SAME easing curve `deck.css`'s panel transitions already use, kept for visual-language consistency — triggered on the full image's `load` event. The LQIP sits behind it throughout (never removed from the DOM/paint), so a slow network simply extends how long the blurred placeholder is visible, never a blank gap.
- **Reduced motion:** no fade — the full image (or LQIP, if not yet loaded) appears via an instant `opacity` swap with `transition: none` under `prefers-reduced-motion: reduce`, matching the project's standing "stop, don't dampen" doctrine (`PITFALLS.md` Pitfall 4, already applied to every other motion element in this scene).

## Spacing/Grid
