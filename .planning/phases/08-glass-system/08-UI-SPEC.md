# Phase 8 — UI Design Contract: Glass System

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.
>
> **Source note:** This is THE glass design contract — the one place all `--glass-*` token values get decided. GLS-03 (verifier re-architecture) and GLS-04 (real-page CPU re-proof) are the arbiters that either confirm or force revision of the numbers below; nothing here ships until both gates pass. ## Glass Grammar

The core contract. Two tiers, two surface classes (panel vs chrome) — never more, per GLASS.md §2.4's "avoid a 3rd/4th elevation tier" guidance and the project's demonstrated token-pruning discipline.

### Recipe table

| Property | Tier-1 panel (optical) | Chrome (header/footer/jump-index) | Tier-2 panel (protected) |
|---|---|---|---|
| Fill | `rgb(255 255 255 / 0.06)` white | `rgb(255 255 255 / 0.05)` white | `rgb(15 18 23 / 0.55)` — **`--panel` itself, translucent**, not white |
| Blur | 12px (Spike-2 default) | 10px (lighter — always-visible, cost-sensitive per GLASS.md §2.4 elevation doctrine) | 10px ("lower-blur" per 08-CONTEXT.md tiering sanction — opacity already does the protecting, less blur needed to sell it) |
| Saturate | 150% | 140% | 130% (less see-through, less desaturation to compensate) |
| Brightness | **0.90** (darkens the captured backdrop *before* the white fill lightens it — see math below; this is the load-bearing number that keeps tier-1 above the 4.5:1 floor) | 0.92 | 0.96 (light insurance only — the dark fill is already protective, doesn't need to fight itself) |
| Applies to | hero, systems, contact (+ the fig-01 *panel wrapper*, framing only) | header, footer, `.deck-index-toggle`, `.deck-jump` (incl. mobile sheet) | experience, patents, skills |
| Token names | `--glass-bg`, `--glass-blur`, `--glass-saturate`, `--glass-brightness` | `--glass-bg-chrome`, `--glass-blur-chrome`, `--glass-saturate-chrome`, `--glass-brightness-chrome` | `--glass-bg-2`, `--glass-blur-2`, `--glass-saturate-2`, `--glass-brightness-2` |

**Blur ceiling (rule, not a token):** 16px is the hard ceiling per Spike-2 (07-SPIKE-GLASS.md — 12→16px costs +0.68pp main-thread, 2.4× the 12px marginal). Nothing in this recipe uses it; it exists as a documented limit for any future adjustment, not a value to consume. Do not add a `--glass-blur-ceiling` token — an unused token violates tokens.css's own pruning discipline. Record the 16px ceiling as a code comment above the glass token block instead (mirrors how the night-sky section documents its "no fifth token" rule in prose, not in an extra variable).

### The 1px top light edge

- **Color/alpha:** `--glass-edge: rgb(255 255 255 / 0.10)` — one token, shared by every glass surface (no per-tier edge variant; the edge is what reads as "optics," it should look identical everywhere for a coherent system).
- **Construction:** a plain `border-top: 1px solid var(--glass-edge)`. **Not** a separate masked DOM element (Comeau's technique, GLASS.md §2.2) — explicitly rejected. That technique doubles the compositing-layer count per glass surface (its own blur pass on the edge strip), and this phase is already at the locked `≤4 simultaneous surfaces` ceiling; doubling it would break the budget rule outright. A flat CSS border costs nothing extra to composite.
- **Does it wrap corners?** **No.** `border-top` only, never a full outline. A single top edge reads as one directional key-light from above (the "cold night air" read 08-CONTEXT.md's Specific Ideas section names); wrapping into the side edges reads as a glow/outline instead, which is a weaker, less directional effect and costs nothing to avoid. Where a surface has `border-radius` (the jump-index pill/list, 6px), the lone top border still renders with correctly rounded top corners — CSS handles this natively, no extra rule needed.
- **Panel-specific nuance (documented, not a bug):** in deck mode, a panel's own DOM top edge (y=0, full-viewport `inset:0`) sits physically *underneath* the fixed header (header is `z-index:20`, panel is `z-index:1` — same screen position). The panel's `border-top` is therefore occluded in deck mode by design; it becomes visible only in classic/scrolling mode, where `.panel` renders as a normal in-flow section with a real top boundary against the sky. The rule stays declared unconditionally (see Elevation/Hierarchy below) for that mode's benefit and for grammar consistency — it is not wasted, just situationally invisible.

### Inner shadow / grain-over-glass — explicitly rejected, both

**Box-shadow / inner-shadow ("seat the glass off the backdrop"):** rejected. GLASS.md §2.1's convention (soft ~20–25% black drop shadow) exists to lift a translucent card off a *light* backdrop; this site's glass floats over an already-dark, already-scrimmed night photo — there is no light surface to separate from, and a shadow would just be a second dark smear stacked on an already-dark scene, adding a filter-chain property for zero perceptual gain. It would also directly contradict the project's own restraint doctrine (08-CONTEXT.md's deferred list: "light-refraction easter eggs → never"). Fill + blur + saturate/brightness + the one top edge is the complete grammar — nothing else.

**Grain-over-glass:** rejected. The prompt asks this explicitly because the sky is banding-proven (07-01's Spike 1: AVIF 10-bit 4:4:4 + baked dither), so the question is legitimate — but glass introduces no *new* gradient surface for banding to reappear in:
- The fill is a **flat, single-alpha color** applied uniformly across the whole surface (`background: var(--glass-bg)`), not a `linear-gradient`/`radial-gradient` — there are no percentage stops to band between.
- `backdrop-filter: blur()/saturate()/brightness()` are per-pixel operations on **already-real, already-dithered photographic data** (the same masters that passed `verify-banding.mjs --selftest`). A multiplicative/convolution filter over real photographic noise preserves the local noise texture; it does not synthesize a new smooth gradient the way a hand-authored CSS gradient would.
- The one place a gradient *could* sneak in — faking the light edge as a soft glow instead of a flat border — is explicitly forbidden above (`border-top: 1px solid`, no gradient, no box-shadow).

No grain layer is added. If a future implementer is tempted to add a radial "sheen" gradient to a glass surface for extra polish, that is a NEW banding risk this spec explicitly does not authorize — re-run `verify-banding.mjs` against it first.

### Classification — text density first (per 08-CONTEXT.md's stated protocol), cross-checked against the measured floor

|---|---|---|---|---|

| fig-01 | N/A — content lives on the solid `.fig` card, never directly over sky | N/A ("no over-sky text" per 07-04-SUMMARY.md gate table) | Exempt (see Per-Surface map) | The one panel with zero contrast exposure regardless of tier, by construction |

| patents | Dense list, prose per entry | **4.58** / 13.94 | **Tier-2** | Density AND tightest measured floor — the two signals agree here |

### Protocol

1. Text density (above) is the **starting classification** — not the final word.
2. GLS-03's screenshot-sampled gate (built as the first task of this phase, per 08-CONTEXT.md's hard precondition) is the **sole arbiter** once real glass values are in place — it measures the actual GPU-composited output, including the real blur kernel, which no hand calculation below can fully replicate.
3. **Any panel failing 4.5:1 worst-case at its assigned tier moves to the next tier up** (tier-1 → tier-2). There is no tier beyond tier-2 in this grammar (GLASS.md §2.4's "avoid a 3rd/4th elevation tier"); if tier-2's protective fill still fails somewhere, that is a stop-and-escalate condition for the orchestrator, not a silent third tier invented mid-phase.
4. Tier-2 is deliberately over-built for headroom (see math below) so it is not expected to need further escalation.

### Expected worst-case math (the delta glass fill adds to the effective scrim)

Baseline (pre-glass, 07-04 measured): worst-case text is `--ink` (`#e8ebef`, relative luminance ≈ **0.829**) against a scrim-composited sky pixel. Back-solving the tightest recorded ratio (**4.58:1** at hero/patents/skills, 1280×800) for the underlying background luminance:

```
4.58 = (0.829 + 0.05) / (L_bg + 0.05)  →  L_bg ≈ 0.142  (≈ sRGB 105/255 — a moderately bright post-scrim sky pixel)
```

**Tier-1 (white fill, the risk case):** a naive 6% white fill laid directly over that pixel with no compensation:
```
new sRGB ≈ 0.06×255 + 0.94×105 ≈ 114  →  L ≈ 0.166  →  ratio ≈ 4.07  — FAILS the 4.5 floor
```
This is why `--glass-brightness: 0.90` exists — it is not decorative, it is the load-bearing compensation. `brightness()` darkens the *captured backdrop* before the fill composites on top of it (filter order: backdrop-filter operates on sampled pixels first, then the element's own background paints over the filtered result):
```
backdrop after brightness(0.90): sRGB 105 × 0.90 ≈ 94.7
then 6% white fill:  0.06×255 + 0.94×94.7 ≈ 104.3  →  L ≈ 0.139  →  ratio ≈ 4.64  — PASSES, ~0.14 headroom
```
Estimated headroom is thin (0.14 vs the 0.08 pre-glass margin) but positive. This is a hand-calculated approximation (sRGB-space compositing, not the browser's actual color-managed pipeline) and does **not** model blur's variance-reduction effect on point-source brightness spikes (stars) — the real screenshot gate may find a healthier or a worse number. Treat 4.64 as "plausible, not proven."

**Tier-2 (dark fill, the protected case):** the same worst-case pixel, `--glass-brightness-2: 0.96` then a 55%-alpha near-`--panel` fill:
```
backdrop after brightness(0.96): sRGB 105 × 0.96 ≈ 100.8
then 55% dark fill (≈ rgb(15,18,23)):  0.55×15 + 0.45×100.8 ≈ 53.6  →  L ≈ 0.036  →  ratio ≈ 10.2 — large headroom
```
This confirms 08-CONTEXT.md's expectation ("values chosen so tier-2 is expected to pass with headroom") — a dark, near-opaque fill functions as a *second scrim layer*, not a competing translucency.

**Chrome risk flag:** the 4.58/6.96 numbers above are measured **only inside the 880px content column** (`verify-contrast.mjs`'s current `.panel` query scope). 07-03's baked column vignette deliberately keeps the **margins brighter** than the column (the galactic core sits "vivid on the right margin" — 07-03-SUMMARY.md) — and the jump-index pill/list sits bottom-right, in exactly that margin. Chrome's real worst-case floor is **unmeasured and could be tighter than any panel number above**. This spec assigns jump-index the standard chrome (tier-1-equivalent, optical) recipe by design intent (08-CONTEXT.md: "smallest, most jewel-like surface, get it right first"), but flags it as the single highest-risk surface in the whole system — if GLS-03's extended chrome sampling (see Per-Surface map) fails here, the protocol in point 3 above applies: move jump-index to the tier-2 (dark, protective) fill, even though that trades away some of its "jewel-like" read. Contrast wins.

## Scrim Interaction

**Decision: KEEP the `.deck::before` scrim unchanged — peak 0.38, same stops, in both deck and classic mode. Not reduced, not retired.**

Rationale, grounded in the math above: the existing scrim gives the tightest panels (hero/patents/skills) only **0.08:1 of headroom** at 1280×800 pre-glass. A translucent **white** tier-1 fill does not "absorb the scrim's duty" the way 08-CONTEXT.md's speculative framing allows for — it actively *fights* the scrim, because white fill lightens the background while the scrim's job is darkening it. The math above shows a naive tier-1 fill alone (no brightness compensation) drops the tightest panel to ~4.07, already below floor, before any scrim reduction is even considered. Reducing the scrim on top of that would be strictly worse. Tier-2's dark fill *does* function as a supplemental scrim (§ math above, ~10.2 headroom) — but the scrim (`.deck::before`) is a single shared layer behind every panel, not something that can be selectively weakened only under tier-2 panels without either duplicating the scrim per-panel (rejected — new DOM/complexity this phase's lock forbids) or accepting reduced protection under tier-1's zero-headroom panels. Given hero has nothing to spare, the shared scrim stays at full strength.

This is the **provisional, conservative default** the actual gate (GLS-03) is free to revise: if the real screenshot-sampled numbers come back with more headroom than this hand math predicts (plausible — blur's variance-reduction on point-source brightness spikes isn't modeled above), a future adjustment could reduce the scrim. This spec does not authorize that speculatively; it locks the safe number now.

**Classic mode:** keeps the identical scrim (already shares the same CSS rule, `html.deck-active .deck::before, html.classic-active .panel`) regardless of the deck-mode decision above — this was already true before Phase 8 and needs no change.

## Elevation / Hierarchy

**Two levels only** (GLASS.md §2.4): chrome (lighter blur/fill, header/footer/jump-index) sits "above" content panels in the sense of being the persistent frame; panels are the featured surface and get marginally more blur/saturation. No third level.

**Active vs. inactive panels — this is the mechanism that keeps the `≤4 simultaneous surfaces` rule true at all times, and it must be implemented exactly as follows:**

Glass CSS (`background`, `backdrop-filter`, `border-top`) is declared **only** on `.panel[data-state="active"]` (deck mode) — never on the bare `.panel` selector and never on `.panel[data-state="inactive"]`. Inactive panels (`opacity:0`, `transform: translateY(24px)`, per the 04-locked transition) simply carry **no** `backdrop-filter` property at all — not `backdrop-filter: none` as an explicit override, just absent. This means:

- Only the one currently-active panel + header + footer + jump-index are ever compositing a blur pass simultaneously — exactly 4, matching Spike-2's tested geometry and the locked ceiling.
- `opacity: 0` on an inactive panel does **not**, by itself, cost anything extra — the GLASS.md §1.4 footgun ("opacity<1 on an ancestor creates a new backdrop root") is about accidentally *exempting sampled content*, not about accidental *cost*; since inactive panels never declare `backdrop-filter` in the first place, there is no backdrop root to worry about on them at all.
- Classic mode (`html.classic-active .panel`) has no active/inactive concept (all 7 panels are simultaneously reachable via scroll) — glass CSS applies unconditionally there. This is the lower-rigor path 08-CONTEXT.md calls out for "a spot-check at both viewports," not a full CPU re-proof; if the spot-check finds a real cost problem, `content-visibility: auto` on off-screen `.panel` sections (GLASS.md §1.4 point 4) is the documented, not-yet-applied mitigation lever.

## Degradation Appearances

### `@supports` absent (no `backdrop-filter` support)

**Zero visual change from today.** This is a stronger and more honest baseline than GLASS.md's suggested `--glass-fallback-bg: var(--panel)` pattern — read directly from `deck.css`/`SiteHeader.astro`/`SiteFooter.astro`, **`.panel`, `header`, and `footer` currently have no background at all** (fully transparent, scrim-only). Only `.deck-index-toggle`/`.deck-jump` currently have `background: var(--bg)`. Therefore:

- Every glass rule (`background`, `backdrop-filter`, `border-top: var(--glass-edge)`) for `.panel`/`header`/`footer` lives **entirely inside** `@supports (backdrop-filter: blur(1px)) or (-webkit-backdrop-filter: blur(1px))`. Outside that block, these three surfaces are declared with nothing new — identical to the live site today.
- `.deck-index-toggle`/`.deck-jump` keep their existing `background: var(--bg)` / `border: 1px solid var(--hair2)` as the unconditional base rule; the glass enhancement (new background + backdrop-filter + new border-top) is added only inside the same `@supports` block, *overriding* the resting `--bg`/`--hair2` values when support exists.
- **No `--glass-fallback-bg` token is needed or declared.** The correct fallback for every surface is its own pre-existing resting style, not a new shared color — this is a deliberate, evidence-based deviation from GLASS.md §5.1's suggestion, made because the research's assumption ("the fallback should look like the current, pre-glass panel design") doesn't match what `.panel`/`header`/`footer` actually look like in this codebase (transparent, not `--panel`-colored).

### `prefers-reduced-transparency: reduce`

Additive pattern (GLASS.md §3.3, Chrome DevRel-sourced): the glass enhancement block above is nested inside `@media (prefers-reduced-transparency: no-preference)` in addition to the `@supports` gate — so both conditions must hold for glass to render. A user who reduces transparency gets exactly the same "zero visual change" baseline as `@supports`-absent, above (same reasoning: no new fallback token, just the existing resting styles). Safari does not support this media feature at all (per GLASS.md §3.3) — Safari users who set the OS-level reduce-transparency preference will still see glass if their Safari version supports `-webkit-backdrop-filter` (it does). This is a documented, accepted gap, not a silent oversight.

### Print

No print stylesheet exists in this codebase yet (confirmed absent in this research pass). Phase 8 adds the minimal rule needed to strip the glass-specific properties only — broader print-quality concerns (the photo, canvas scene) are out of scope here:

```css
@media print {
  .panel, header, footer, .deck-index-toggle, .deck-jump {
    background: none;
    backdrop-filter: none;
    -webkit-backdrop-filter: none;
    border-top: none;
  }
}
```

## Motion

- **Glass filter properties themselves never animate.** No `transition` on `background`, `backdrop-filter`, blur radius, saturate, or brightness, on any surface, ever. Glass is a static, binary treatment — present or absent per the `@supports`/`data-state`/media-query gates above, never faded/tweened.
- **Panel position transitions stay transform-only** by inheritance from 04-UI-SPEC.md (translateY 24px, 420ms, `cubic-bezier(0.16, 1, 0.3, 1)`) — this phase does not touch `deck.css`'s transition mechanics.
- **The backdrop-root trap, named explicitly:** the 04-locked panel transition *also* animates `opacity` (0→1 incoming, 1→0 outgoing) simultaneously with the transform. Because the incoming panel's `data-state` flips to `"active"` — and therefore gains `backdrop-filter` — at the *start* of that same transition, the browser must re-run capture→blur→composite on every animation frame of the opacity ramp (~25 frames at 420ms/60fps). **Opacity-animating a glass surface is forbidden as a *new* pattern this phase — but this specific case is a pre-existing 04-locked mechanic, not something Phase 8 introduces.** It is accepted here as a bounded, user-triggered cost (one navigation event, not an idle/ambient loop) — Spike-2's 60s idle soak did not measure it, so GLS-04's real-page re-proof should note whether navigation-time frames show any jank, but fixing it (e.g., deferring `backdrop-filter` until a `transitionend`-driven class swap) is out of scope for this phase — it would touch `deck.ts`'s locked architecture.
- **Reduced motion:** unaffected by glass. `prefers-reduced-motion: reduce` already collapses the panel transition to `transition: none` (04-locked, unchanged) — with no opacity ramp at all, the backdrop-root trap above doesn't apply under reduced motion; panels snap directly between static glass/no-glass states with a single, cheap re-render.

                                       Tiering Assignment math */
