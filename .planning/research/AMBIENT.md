# v3.0 "Real Sky" — Ambient Systems Research (Technique + Budget)

**Lane:** Technique + per-system frame budget + composition doctrine for the four locked ambient animations.
**Researched:** 2026-07-18
**Confidence key:** HIGH (official docs / MDN / cross-verified), MEDIUM (single strong source, dev blog, or perf writeup not independently re-verified), LOW (inference / no direct source found — flagged explicitly, treat as a hypothesis to validate with a prototype)

**Existing engine ground truth (read directly from `src/lib/nightsky/`, not researched):**
- One rAF loop lives in `scene.ts`'s `tick()`; Layer 0 (starfield/Milky Way) is generated ONCE per resize into an offscreen canvas and blitted every frame via `drawImage()` — no per-frame regeneration of Layer 0 is allowed by the current architecture.
- Pause machine already exists: `prefers-reduced-motion` (`matchMedia('(prefers-reduced-motion: reduce)')`) short-circuits the loop entirely and renders a single static frame; tab-hidden and fig-01-panel-active also gate the loop off.
- No `backdrop-filter` usage exists yet anywhere in the codebase — the "glass layer" the prompt references is being introduced by a parallel lane in this same milestone. This document flags the compositing interaction but does not resolve it (per instructions, that's a cross-lane coordination item).

## 1. Clouds / Haze Layers

### Technique recommendation: canvas sprites blitted into the existing Layer-0/Layer-2 pipeline, NOT CSS `<div>` layers

**Reasoning:**
- The whole point of the current architecture is one rAF loop with strictly bounded per-frame canvas work sitting under a single composited surface. Introducing a second rendering system (CSS-animated `<div>` layers) for clouds means a second thing to pause/reduced-motion-gate, a second z-index/stacking-context concern relative to the photo `<img>`/canvas and the incoming glass panel, and — critically — the backdrop-filter interaction below. Keeping clouds as canvas draws inside the existing `tick()` keeps ONE pause machine and ONE compositing surface.
- Cost profile is cheap and well precedented: pre-render 2–3 large semi-transparent cloud/haze sprites ONCE (either at build time as PNG/WebP assets, or generated once into an offscreen canvas like Layer 0 already does), then every frame just `drawImage()` them at a slowly-incrementing x-offset with wraparound (the classic side-scroller infinite-background technique — draw the sprite, then draw a second copy offset by its own width so the seam never shows, advance both by `speed * dt`, reset the offset modulo sprite width). This is the standard technique documented for canvas parallax backgrounds. (MEDIUM — synthesized from multiple canvas-game perf writeups, technique is old and uncontested: [Optimizing HTML5 Canvas — Code Theory](https://codetheory.in/optimizing-html5-canvas-to-improve-your-game-performance/), [HTML5 Canvas Game: Panning a Background](https://blog.sklambert.com/html5-canvas-game-panning-a-background/), [MDN Optimizing canvas](https://developer.mozilla.org/en-US/docs/Web/API/Canvas_API/Tutorial/Optimizing_canvas))
- `drawImage()` of a pre-rendered bitmap is GPU-texture-copy-cheap regardless of whether the source is canvas-generated or an `<img>`-decoded asset — MDN's own canvas optimization guide explicitly recommends pre-rendering similar-looking objects offscreen once instead of recomputing gradients/shapes per frame (HIGH — [MDN Optimizing canvas](https://developer.mozilla.org/en-US/docs/Web/API/Canvas_API/Tutorial/Optimizing_canvas)). This is the exact pattern the codebase already uses for Layer 0 — clouds should be "Layer 1," a second pre-rendered-once-per-resize offscreen bitmap, blitted with a translating source/dest rect instead of a static one.

### Layering / tiling strategy
- **2 sprite layers is enough for convincing depth**, 3 at most: a far high haze layer (very low opacity, very slow, large soft-edged blobs) and a near cloud layer (slightly higher opacity/contrast, faster). A third thin low-horizon haze band can be folded into the same Layer-0 static bake instead of animating, since horizon haze barely moves visually at cloud-deck distance. Two independently-animated layers is the standard minimum for a parallax illusion — one layer alone reads as "sliding texture," not depth (MEDIUM — general parallax-layering consensus across game-dev sources found; no single canonical citation, but every source surveyed used ≥2 layers for the effect to register: [Lumitree parallax blog](https://lumitree.art/blog/parallax-effect), [Justin Jerzak canvas parallax tutorial](https://justinjerzak.wordpress.com/2014/05/14/javascript-parallax-scroll-canvas-tutorial/)).
- **Tiling:** author each cloud sprite as a horizontally-seamless tile (left edge alpha-matches right edge) sized ~1.5–2x max viewport width, so only ONE wraparound draw pair is ever needed per layer per frame (draw once at `x`, once at `x - spriteWidth` or `x + spriteWidth`, whichever is about to enter frame) — same technique the sklambert canvas-panning writeup describes (MEDIUM — [blog.sklambert.com](https://blog.sklambert.com/html5-canvas-game-panning-a-background/)).
- **Round translate coordinates to integers before `drawImage()`** — sub-pixel positions force the browser into a more expensive bilinear resample path; multiple canvas-perf sources flag this explicitly (MEDIUM — [codetheory.in](https://codetheory.in/optimizing-html5-canvas-to-improve-your-game-performance/), [jaredwilli canvas perf gist](https://gist.github.com/jaredwilli/5469626)).

### Speeds and opacity ranges
- No canonical numeric standard exists for "realistic" cloud drift speed in a stylized night sky (this is an art-direction call, not a physics one) — treat the following as a **starting point to tune by eye, LOW confidence on the specific numbers, HIGH confidence on the ordering/ratio logic**:
  - Far haze layer: ~2–4 px/s at 1x DPR (imperceptible as "motion," reads only as slow gradient shift over 30+ seconds on screen).
  - Near cloud layer: ~8–15 px/s (visible drift, still calm — real high-altitude cirrus/haze at a stylized "observing from a dark-sky field" register, not storm clouds).
  - Ratio between layers should be roughly 1:3 to 1:5 (far:near) to sell parallax depth without needing a third layer.
- **Opacity ceiling is the binding constraint, not speed.** Because Layer 0 already carries the full star density and the Milky Way band, cloud sprites must stay low-alpha (rough starting range 0.04–0.12 peak alpha for haze, up to ~0.18 for the near cloud layer at its densest point) so they read as atmospheric depth, not overcast — and so text contrast (WCAG legibility is already a locked constraint per the honesty/Lighthouse gates) isn't affected, since UI copy sits in front of/above this layer, not blended through it. This is a design-tuning judgment call, not sourced — flag for visual QA once real photo base is in.

### CSS-on-compositor vs canvas — and the backdrop-filter interaction (flag, not resolved)
- **General web-perf doctrine (HIGH confidence, well-established):** `transform` and `opacity` are the only two CSS properties that can be animated compositor-only — no layout, no paint, just a GPU compositing step ([web.dev / web fundamentals: Stick to Compositor-Only Properties](https://developers.google.com/web/fundamentals/performance/rendering/stick-to-compositor-only-properties-and-manage-layer-count), [web.dev animations-guide](https://web.dev/articles/animations-guide)). A CSS `<div>` cloud layer animated via `transform: translateX()` keyframes would, in isolation, be extremely cheap — arguably cheaper than canvas `drawImage()` since it moves work off the main thread entirely onto the compositor thread.
- **However, this project already sits under a `backdrop-filter` glass panel design (per project conventions/CLAUDE.md tokens) that a parallel lane in this milestone is introducing.** This is the critical interaction to flag:
  - `backdrop-filter` works by having the browser **re-render/re-composite everything visually underneath the filtered element, every time that content changes, then re-run the blur.** This is explicitly documented as the reason `backdrop-filter` is more expensive than a regular `filter`: "the browser has to render the entire scene behind the element first, then apply the filter, then composite the element on top" (MEDIUM — [MDN backdrop-filter](https://developer.mozilla.org/en-US/docs/Web/CSS/backdrop-filter), cross-referenced against [OpenReplay blog on backdrop-filter](https://blog.openreplay.com/creating-blurred-backgrounds-css-backdrop-filter/)).
  - Whether the underlying layer is a `<div>` with `transform: translateX()` or a `<canvas>` with `drawImage()` calls, **if it sits visually behind a `backdrop-filter` element, every visual change to it forces that backdrop-filter region to re-blur on the next composite** — a continuously-translating cloud layer under a glass panel means the glass panel's blur is being recomputed every frame it overlaps, not once. There is direct precedent for this being a real, documented performance problem: Firefox has an open, tracked bug for `backdrop-filter` blur being laggy specifically "when many elements are rendered" behind/near it ([Mozilla Bugzilla #1718471](https://bugzilla.mozilla.org/show_bug.cgi?id=1718471)), and community reports (GSAP forums, various GitHub issues) describe backdrop-filter regions becoming the bottleneck specifically when the content behind them is animating (MEDIUM — [GSAP forum thread](https://gsap.com/community/forums/topic/25362-browser-performance-for-animation-backdrop-filter/)).
  - **This means: cloud layer choice (canvas vs CSS) does NOT avoid this cost either way** — the cost is triggered by "does the glass panel's backdrop region overlap animating pixels," not by which technology drew those pixels. Two mitigations exist, both requiring the glass lane's input to actually land on: (a) keep the glass panel's backdrop-filtered footprint small/localized (e.g. a card, not a full-bleed strip) so the re-blur cost is bounded by a small pixel area rather than viewport-width; (b) if the glass panel sits over a region where clouds are visible, consider excluding that specific z-order slice from the cloud layer's paint region, or accept the added GPU cost there as bounded because the region is small. **Flagging for the glass-lane researcher/implementer: confirm the glass panel's footprint size and whether it's likely to overlap the animated cloud layer's visible area, since that overlap area — not the cloud technique itself — is what determines the actual added cost.**
  - Given the above, and given the project's existing single-rAF-loop, single-compositing-surface discipline, **canvas is still the recommended default** even though CSS-transform-only would in isolation be cheaper: it keeps clouds inside the existing pause machine (no separate `animation-play-state` wiring to keep in sync with the rAF pause gates), keeps one source of truth for "is anything animating right now," and the backdrop-filter re-blur cost applies regardless of which technique is chosen, so CSS's isolated-cheapness advantage doesn't actually materialize here.

**Verdict:** Canvas sprite layers (2, matching the existing Layer-0-style pre-render-once + blit pattern), integer-rounded translate, opacity capped low, drawn as part of the existing `tick()`. Flag the backdrop-filter/cloud-overlap interaction to the glass lane; do not let cloud sprites animate under/near the glass panel's footprint without confirming the re-blur cost is bounded.

## 2. Parallax on Panel Change

### Technique recommendation: CSS `transform` transitions on layered elements, triggered once per discrete panel-change event — NOT continuous canvas camera-offset math

**Reasoning:**
- The trigger here is fundamentally different from clouds/aurora/scintillation: it's a **discrete, one-shot event** (wheel gesture → panel change), not a continuous per-frame animation. This is exactly the case CSS transitions are built for, and it's the cheapest possible implementation: set a `transition: transform 300-500ms <easing>` on the sky/horizon/foreground layer elements, then on panel change simply update each layer's `transform: translate3d(x, y, 0)` target value (different per layer for the parallax rate) and let the browser's compositor interpolate — **zero JS per-frame work, zero rAF involvement, the compositor thread owns the whole transition** (HIGH — same compositor-only-properties doctrine as above: [web.dev animations-guide](https://web.dev/articles/animations-guide), [web fundamentals: compositor-only properties](https://developers.google.com/web/fundamentals/performance/rendering/stick-to-compositor-only-properties-and-manage-layer-count)).
- This is a materially better fit than driving the shift through the canvas/rAF loop (a "camera offset" approach where `tick()` interpolates an offset and redraws every frame for the transition's duration): that would mean spinning up per-frame canvas redraws for 300-500ms on every single panel change, which is exactly the kind of continuous work the "occasional single beam / occasional single meteor" bounded-work philosophy in the existing engine tries to avoid for anything that isn't already inside the steady-state per-frame budget. A CSS transition achieves the same visual result with the browser doing the interpolation off the main thread.
- **Layered-element structure:** three (or however many) stacked elements/layers — sky (photo base + Layer 0 canvas), horizon, foreground — each get a different `translate` distance for the same panel-change event, classic parallax-depth-via-differential-transform, same technique surveyed for scroll-based parallax but applied to a **discrete state change instead of scroll position** (MEDIUM — technique is a direct adaptation of standard CSS parallax layering: [Medium: Multi-Layered Parallax Illustration with CSS & JS](https://medium.com/@patrickwestwood/how-to-make-multi-layered-parallax-illustration-with-css-javascript-2b56883c3f27), [freefrontend CSS parallax collection](https://freefrontend.com/css-parallax/)). Foreground moves most, horizon a middle amount, deep sky least (or is held static/moves only fractionally) — inverse of distance, standard parallax convention.
- **300–500ms is a reasonable transition length** and aligns with typical "meaningful but not sluggish" UI transition timing; no single authoritative source pins an exact number for this specific use case, treat as a design-tuning parameter within the requested range (LOW confidence on the exact ms figure — it's a design choice within the caller's own stated bound, not something to source-verify).
- **Reduced-motion = instant:** gate the transition itself, not just future frames — under `prefers-reduced-motion: reduce`, either omit the `transition` property entirely (snap to the new transform value in one frame) or explicitly override to `transition: none` inside an `@media (prefers-reduced-motion: reduce)` block. This is the documented, WCAG-referenced standard pattern (HIGH — [W3C WAI technique C39](https://www.w3.org/WAI/WCAG22/Techniques/css/C39), [web.dev prefers-reduced-motion](https://web.dev/articles/prefers-reduced-motion), [MDN prefers-reduced-motion](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-reduced-motion)) — and it's exactly consistent with the existing codebase's own reduced-motion doctrine already documented in `scene.ts` (single static frame, no animated path at all).
- **No continuous scroll-linked work** falls out naturally from this approach: because the trigger is the existing discrete panel-change event (not a scroll or wheel-delta listener driving per-pixel updates), there's no scroll-jank risk and no need for a `scroll`/`wheel` event handler doing per-frame transform math — the panel-change code path presumably already exists (it drives which panel is active today) and this just adds a transform-target update alongside it.

**Verdict:** CSS `transform: translate3d()` transitions (300-500ms) on 2-3 stacked layer elements, values updated once at the moment of each discrete panel-change event, `prefers-reduced-motion` snaps instantly (no transition). Zero rAF/JS-per-frame cost — the compositor thread carries the whole animation.

## 3. Aurora Shimmer

### Technique recommendation: canvas-painted curtains via layered translucent sine/noise waves, updated at a throttled rate (not full 60fps) — not CSS filter tricks, not full WebGL

**Options surveyed:**
1. **Pure CSS gradient/filter animation** (animating `background-position`, `filter: hue-rotate()`, or blurred gradient blobs via keyframes) — cheap (compositor-only if restricted to transform/opacity-driven gradient position tricks) but visually reads as "moving blob," not the specific curtain/undulation character of aurora; multiple CSS-only aurora demos exist (a "CSS Aurora effect" DEV Community post was found: [dev.to/oobleck/css-aurora-effect-569n](https://dev.to/oobleck/css-aurora-effect-569n)) confirming it's achievable but the visual fidelity ceiling is lower than a noise-driven curtain shape (LOW-MEDIUM — didn't fetch the full CSS implementation to verify quality, judging from technique description only).
2. **Canvas 2D, layered translucent sine-wave "curtains" with varying speed/amplitude/hue per layer** — the dominant pattern found across working implementations: multiple example projects (CodePen "Aurora Borealis Canvas Animation" using simplex noise, and a documented open-source implementation) independently converge on **2-3 translucent wave/curtain layers**, each with independently varying frequency, amplitude, and color, alpha-blended together (MEDIUM — [Btsan/Aurora-Effect GitHub](https://github.com/Btsan/Aurora-Effect), [CodePen vroom/kKdmBe](https://codepen.io/vroom/pen/kKdmBe), summarized via [Grokipedia Aurora Borealis Simulation](https://grokipedia.com/page/Aurora_Borealis_Simulation_in_HTML)).
3. **Full WebGL/shader approach** (three.js aurora shader examples exist: [CodePen favina/bemzVK](https://codepen.io/favina/pen/bemzVK)) — visually the highest ceiling (proper procedural noise curtains with real vertical light-ray falloff) but introduces a second rendering context (WebGL alongside the existing 2D canvas), a shader compile step, and a GPU dependency profile that doesn't match "keep everything inside the existing bounded 2D canvas pipeline." Given this is meant to be a **faint, low-luminance ambient touch** (per the brief) rather than a hero visual, the fidelity gap doesn't justify the architectural cost. Not recommended for this project.
4. **Pre-rendered frame-blended sprite animation** (author N pre-rendered aurora frames, cross-fade between them) — cheapest possible runtime cost (pure `drawImage()` + alpha crossfade, no per-frame math) but least "alive"-feeling unless N is fairly large (8-16+ frames) to avoid visible looping/stepping, and authoring that many hand-made or pre-generated frames is more production overhead than a small procedural function. Worth keeping as a fallback for the mobile degradation ladder (see §5) since it's the cheapest option, but not the primary technique.

**Recommended technique:** Canvas 2D, 2 translucent curtain layers using a cheap **value-noise** (not gradient/simplex/Perlin) horizontal displacement function for the undulation, each layer a vertical gradient (bright-to-transparent) clipped to a wavy top edge computed from the noise function, composited with `globalCompositeOperation = 'lighter'` or plain alpha blending for the characteristic glow-additive look. Value noise is explicitly called out by an existing implementation as the deliberate choice **because it's computationally cheaper than gradient noise while still producing acceptable smooth transitions** — the source implementation's author explicitly traded a known minor visual artifact (occasional visible horizontal banding, "value noise isn't smooth") for the performance win (MEDIUM — [Btsan/Aurora-Effect README](https://github.com/Btsan/Aurora-Effect)).

### Noise source: precomputed table, not runtime generation
- Given the "low-fps layer" budget constraint from the brief, the noise driving the undulation should be a **small precomputed lookup table** (e.g. a few hundred value-noise samples generated once at init, or even a simple array of precomputed sine-sum values with slightly mismatched frequencies to avoid obvious periodicity), sampled/interpolated at runtime rather than computing a noise function fresh every update. This is not a claim from a specific aurora source but a direct application of the same "pre-render once, cheap sample at runtime" doctrine MDN documents generally for canvas performance (HIGH confidence on the general principle — [MDN Optimizing canvas](https://developer.mozilla.org/en-US/docs/Web/API/Canvas_API/Tutorial/Optimizing_canvas); MEDIUM on applying it specifically to aurora, since no aurora-specific source discussed lookup-table noise explicitly).

### Update-frequency budget: yes, a low-fps layer is viable and recommended
- Aurora undulation is, physically and visually, a **slow** phenomenon (curtains shift over multiple seconds, not frame-to-frame) — there is no perceptual cost to updating the underlying noise/shape state only every 3-5 frames (i.e., ~12-20Hz at a 60fps target) while still redrawing every frame if needed for compositing, OR — cheaper still — only redrawing the aurora canvas region itself every 3-5 frames and leaving the pixels untouched on skipped frames (since nothing else needs to composite through that exact region at 60fps if it's a background layer behind the stars). This is a direct, uncontested application of the general "decouple update rate from render rate" / manual frame-skipping pattern documented for canvas+rAF (MEDIUM-HIGH — technique itself is well-established: [Qubits & Bytes: Control the Frame Rate of rAF Callbacks](https://qubitsandbytes.co.uk/javascript/control-the-frame-rate-of-requestanimationframe-callbacks/), [GeeksforGeeks: control fps with rAF](https://www.geeksforgeeks.org/how-to-control-fps-with-requestanimationframe/) — general rAF-throttling pattern, not aurora-specific, but directly transferable).
- Practically: keep aurora's own state-update function (advancing the noise phase, recomputing curtain shape/color) gated behind a frame counter (`frameCount % 4 === 0`), but note the *drawing* still needs to happen every rAF tick if the aurora canvas layer needs to composite under a full-frame redraw (e.g. if it's baked into the same canvas as Layer 0/2 rather than a separate compositor layer) — the saving is in the noise-math, not necessarily the blit. If aurora is instead its own canvas element (separate compositing layer, `drawImage`'d or CSS-composited), then genuinely skipping the redraw itself on non-update frames is possible and cheaper still.

### Luminance ceiling — "the one sanctioned light source"
- No external source addresses this specific design constraint (it's a project-specific art-direction/premise decision, not a researchable technical fact) — this is a **design recommendation, LOW confidence / inference only**: keep aurora peak alpha materially below the brightest stars' rendered alpha and below the Milky Way band's peak brightness in Layer 0 (sample/compare against whatever alpha values `starfield.ts`/`tokens.ts` currently use for the Milky Way band, since that's the current brightest ambient feature, and cap aurora below it — e.g. aurora peak alpha at ~50-70% of the Milky Way band's peak alpha as a starting ratio, tuned by eye). Because aurora is later-composited as a translucent overlay rather than baked into Layer 0's fixed bitmap, its resolvable luminance ceiling is fully controllable via a single alpha multiplier at draw time, making this an easy constraint to enforce mechanically once the target ratio is picked in a design pass.

**Verdict:** Canvas 2D, 2 translucent curtain layers, value-noise-driven wavy top edge from a small precomputed lookup table, noise-phase updates throttled to every 3-5 frames (draw/composite can stay at full rate or also throttle if aurora is isolated to its own layer), peak alpha capped below Layer 0's Milky Way band brightness via a single tunable multiplier.

## 4. Scintillation (Atmospheric Star Twinkle)

### Physical basis (grounding the technique, not implementing real physics)
- Real stellar scintillation is caused by turbulent atmospheric layers refracting starlight; the temporal frequency of brightness fluctuation can reach up to ~50Hz in extreme cases (driven by turbulent-cell transit at typical wind speeds), and — critically for the "correlated color shift" requirement — **shorter wavelengths (blue) scintillate more strongly than longer ones (red)**, which is why bright, low-altitude stars like Sirius visibly flash blue/green/red to the naked eye (HIGH confidence on the physical facts — [BBC Sky at Night: Why do stars twinkle?](https://www.skyatnightmagazine.com/space-science/why-do-stars-twinkle), [Astronomy.com: Simply Scintillating](https://www.astronomy.com/science/simply-scintillating/), cross-referenced against [arXiv: Simulation of optical interstellar scintillation](https://arxiv.org/abs/1301.0514)). The rigorous scientific simulation approach (FFT-based fractal cloud refraction + Fresnel integrals) is explicitly overkill for a decorative web animation — that's simulating the physical cause, not the visual effect — and is not recommended (HIGH confidence this is the wrong tool for this job, by inspection: that's a research-grade optical-physics simulation, not a rendering technique).
- **Recommended simplification:** don't simulate the atmosphere; simulate the *visible symptom* directly with a lightweight per-star oscillator: a slow, semi-random brightness modulation (not truly 50Hz — that's imperceptible/aliased on a 60fps display anyway and would just look like static noise; a perceptible "twinkle" cadence is much lower, on the order of a few Hz of apparent variation) layered with an occasional, brief, low-probability color-channel nudge on the small subset of the brightest/lowest-altitude-coded stars only (mirroring the real-world fact that only bright stars near the horizon show visible color scintillation — most stars just vary in brightness).

### Precomputed tables vs runtime noise
- Same doctrine as aurora: use a **small precomputed per-star phase-offset + frequency table** (e.g., each twinkle-eligible star gets a fixed random phase and frequency assigned once at Layer-0-generation time, stored alongside its other precomputed properties) and drive brightness at runtime with a cheap `sin(phase + t * frequency)` sample (or an even cheaper precomputed lookup array indexed by frame count, avoiding a `Math.sin()` call per star per frame if profiling shows it matters) rather than any runtime noise-function evaluation. This is directly consistent with the existing codebase's own pattern — it already runs "twinkle subset ~40 stars" per the brief, meaning this exact architecture (a bounded twinkle-eligible subset with precomputed per-star parameters) is already how the current engine works; scintillation is an enhancement to that existing subset's *update function*, not a new subsystem. (No external source needed here — HIGH confidence this is the correct integration point, based on directly reading the existing engine's documented behavior.)
- **Color shift implementation:** rather than true wavelength-dependent refraction math, a cheap approximation is sufficient — on a low-probability trigger (e.g., only for the top few brightest stars in the twinkle subset, and only occasionally, not every frame), nudge the star's rendered RGB slightly toward blue/white or add a faint red/orange fringe for one or two frames, then relax back. This is a stylized reference to the real phenomenon, not a physical simulation (LOW-MEDIUM confidence this specific implementation shape is "correct" — it's a reasonable design interpretation of the sourced physical fact, not itself sourced from an existing twinkle-color-shift implementation, since no such implementation writeup was found in this search pass).

### Budget: how many stars can scintillate
- The brief states the existing engine already twinkles "~40 stars" per frame as its bounded subset, and the measured baseline (5.6-6.9% idle CPU) already includes that cost. Scintillation as specified here is a *replacement* of the existing simple twinkle for that same ~40-star subset (richer per-star modulation curve + occasional color nudge on a handful of the brightest of those 40), not an increase in star *count* — so the recommended default is **keep the twinkle-eligible subset size the same (~40)** and spend the added budget on per-star modulation quality (a slightly richer waveform than pure sine, e.g. sum of 2 mismatched-frequency sine samples from the precomputed table for a less mechanical cadence) rather than widening the subset. Widening the subset is the lever to cut first if the budget table in §5 shows pressure (LOW-MEDIUM confidence — this is a budget-allocation recommendation grounded in the stated existing baseline, not an external source).

### Precedent: is it acceptable that the photo's own baked-in stars don't twinkle while only the drawn overlay stars do?
- No canonical design-precedent source directly answers this exact question (it wasn't discussed in any survey or perf writeup found), but there is a strong **commercial/industry precedent by direct analogy**: the entire "night sky overlay" motion-graphics product category (Envato Elements, Adobe Stock, etc. — multiple listings found: [Envato "Twinkling Starry Night Sky with Subtle Clouds"](https://elements.envato.com/night-sky-and-stars-SZUPK8F), [Adobe Stock "Twinkling Stars" search](https://stock.adobe.com/search?k=twinkling+stars)) is built entirely on the premise of compositing an animated twinkling-star layer **over** static or live-action footage that has its own baked-in (non-animated) stars/background — that's the explicit intended use case of a "stars overlay" product: layer it on top of an existing photo/video plate. This means the asymmetry this project is proposing (photo base with fixed stars + drawn animated stars on top) is not a novel or untested composition — it's the standard structure of an entire existing content category (MEDIUM confidence via analogy — direct precedent for the *compositing pattern*, not a direct statement that viewers don't notice or mind the asymmetry, which wasn't independently verified).
- **Practical mitigation regardless:** the photo base should be curated/processed (already implied by other lanes of this milestone) so its own visible stars are relatively few/dim/soft compared to the crisp, precisely-placed drawn star layer — minimizing the number of "static" stars a viewer's eye can individually track and compare against a twinkling neighbor. This is a photo-selection/processing recommendation for the imagery lane, not something this document can resolve — **flag for the photo-sourcing lane**: prefer a base photo with a fainter/softer star presence (e.g., slight motion-blur or long-exposure star-trail-free but soft-focus stars) so the drawn overlay stars visually "own" the crisp-twinkle role and the photo's stars read more as ambient texture/Milky Way glow than as individually comparable points.

**Verdict:** Enhance the existing ~40-star twinkle-eligible subset's update function with a richer precomputed-phase waveform (2-oscillator sum) and an occasional brief color nudge limited to the brightest few stars in that subset; do not widen the subset size as the default. Photo-base star softness is a cross-lane flag, not resolved here.

## 5. Composition Doctrine — Bounded Ambient Set

### Evolving the pause machine
The existing pause machine (`{tab hidden, fig-01 panel active, reduced-motion → single static frame}`) extends cleanly — none of the four new systems need a *new* pause condition beyond what already exists, since they all live inside or alongside the same single rAF loop and the same reduced-motion gate:

| Trigger | Current behavior | v3.0 behavior (all 4 systems) |
|---|---|---|
| Tab hidden | rAF loop stops | Same — loop stop halts clouds/aurora/scintillation; parallax has no pending transition to interrupt (CSS transitions on a hidden tab simply don't matter — no repaint occurs) |
| Fig-01 panel active | rAF loop stops | Same — all four ambient systems freeze; this is actually the natural place clouds/aurora "pause" without needing bespoke logic, since they're driven by the same `tick()` |
| `prefers-reduced-motion: reduce` | Loop never starts; single static frame rendered | Loop never starts (clouds/aurora/scintillation render their first-frame/steady-state look baked into that one static frame, no motion); parallax transitions are removed via `@media (prefers-reduced-motion: reduce) { transition: none }` so panel changes snap instantly instead of animating |

No new pause states are needed — this is a meaningful finding in itself: **the existing three-condition pause machine already fully covers all four new systems** as long as clouds/aurora/scintillation are implemented as per-frame work inside the existing `tick()` (not separate independently-running timers), and parallax is implemented as CSS transitions gated by the same reduced-motion media query the rest of the site presumably already respects.

### Per-system frame-cost model
Precise cycle-level costs aren't independently benchmarkable without a working prototype (flagging this honestly — the numbers below are relative-cost estimates grounded in the *technique choices* recommended above and the general canvas-perf literature cited per-section, not measured). Framed as **relative addition to the existing measured 5.6-6.9% idle CPU baseline**:

| System | Technique | Per-frame work | Relative cost vs. baseline | Confidence |
|---|---|---|---|---|
| Clouds/haze | 2x pre-rendered sprite `drawImage()` w/ integer-rounded translate | 2 blits/frame, no per-frame math beyond incrementing 2 floats | Low — comparable to or cheaper than the existing single-beam/meteor draws, since it's pure blit (no path/gradient computation) | MEDIUM |
| Parallax (panel change) | CSS `transform` transition, compositor-thread only | Zero JS/canvas work; browser compositor interpolates for 300-500ms only on the (infrequent) panel-change event | Negligible — effectively free relative to the rAF loop, since it's off-main-thread and event-triggered, not continuous | HIGH (compositor-only-properties doctrine is well-established) |
| Aurora shimmer | Canvas curtain layers, noise-phase updates throttled to every 3-5 frames | ~1 update-math pass every 3-5 frames + a redraw (full or partial) each frame | Low-moderate — the throttled update keeps the expensive part (noise sampling) infrequent; the redraw itself is a gradient-clipped fill over a small vertical band, cheaper than a full-viewport operation | MEDIUM |
| Scintillation | Enhanced per-star waveform on existing ~40-star subset, precomputed tables | Same star COUNT as today's twinkle, marginally richer per-star math (2-oscillator sum instead of 1, occasional color nudge on a handful) | Negligible-to-low — this is explicitly framed as a quality upgrade to existing bounded work, not new work at a new scale | HIGH (grounded directly in existing engine's documented ~40-star bound, not external research) |

**Conclusion on the <10% idle CPU floor:** given the existing baseline of 5.6-6.9% already includes twinkle (40 stars), fireflies (9), occasional beam/meteor, and a full Layer-0 blit every frame, and given that three of the four new systems (clouds, aurora, scintillation-as-enhancement) are designed specifically to be blit-dominated or throttled rather than adding new per-frame computational classes, and the fourth (parallax) is compositor-only and event-triggered rather than continuous — **all four systems fitting within the <10% floor is plausible without requiring a system to be dropped outright**, provided: (a) clouds stay at 2 layers not 3+, (b) aurora's noise-phase update stays throttled (not full-rate), (c) scintillation doesn't widen the star subset, and (d) the backdrop-filter/cloud-overlap interaction flagged in §1 doesn't turn out to add unexpectedly large re-blur cost once the glass lane's actual footprint is known. **This last point is the one genuine unresolved risk to the budget conclusion** — it's the one cost this document cannot bound without input from the glass-panel lane, since backdrop-filter re-composite cost scales with the *glass panel's own pixel footprint*, a variable this researcher doesn't control. Recommend a real-device CPU profile pass (not just estimation) once clouds + glass panel are both implemented, specifically checking whether cloud-layer animation visible under/near the glass panel changes the CPU number materially versus clouds animating with the glass panel absent from view.

### Mobile / low-power degradation ladder
No project-specific mobile CPU budget data exists yet to size this precisely (flagging as LOW confidence on exact thresholds — this is a reasonable ladder shape, not benchmarked numbers), but the general pattern of "detect constrained context, drop expensive-first" is standard practice; general sources on backdrop-filter specifically note mobile blur budgets degrade faster than desktop (MEDIUM — [F22 Labs: How CSS Properties Affect Website Performance](https://www.f22labs.com/blogs/how-css-properties-affect-website-performance/), noting blur radii above ~20px and multiple simultaneous blur regions both push mobile GPU cost up sharply; OpenReplay backdrop-filter piece notes practical device handling tends to degrade past a handful of concurrent blur regions). Recommended ladder, cheapest-to-cut first, most-essential last:

1. **Full desktop/high-power:** all 4 systems active as specified (2-layer clouds, full parallax, throttled aurora, enhanced scintillation).
2. **Mid-tier / small screens (viewport width below a chosen breakpoint, e.g. tablet-and-down):** drop clouds to 1 layer (near layer only, cut the far haze layer — least individually perceptible loss since it's the lowest-alpha, slowest-moving layer already); keep aurora, parallax, scintillation as-is.
3. **Low-power mobile (detect via a practical proxy — `navigator.hardwareConcurrency` low core count, or simply "narrow viewport" as the pragmatic signal already used by most responsive designs, since `navigator.deviceMemory`/battery APIs have poor/inconsistent browser support and shouldn't be load-bearing):** drop aurora's per-frame redraw further (e.g., throttle to every 6-8 frames instead of 3-5, or reduce to a single curtain layer instead of two); keep clouds at 1 layer; keep parallax (it's compositor-only and cheap regardless of device, no reason to cut); keep scintillation but consider narrowing the color-nudge feature (brightness-only twinkle, skip the color-shift flourish) since it's the most visually-secondary refinement of the four.
4. **Universal floor, all tiers:** parallax is never cut (cheapest system, compositor-only, and arguably the most structurally important since it's tied to the core panel-navigation interaction, not decorative ambiance) and reduced-motion always wins outright over every tier (full stop, single static frame, as today).

This ladder deliberately protects parallax at every tier (cheapest + most functionally tied to navigation) and treats aurora as the most gradually-degradable system (throttle before cut, cut layer-count before cutting entirely) since it's explicitly framed in the brief as "faint" and a low-fps layer is already its native design, making it the natural dial to turn down first among the three canvas-continuous systems.
